# Лабораторна робота 2: Шарова архітектура та доменна модель

## Що змінилося в структурі проєкту порівняно з лабораторною 1?

У першій лабораторній не було ніяких вимог з приводу архітектури, тому вся логіка ендпоінта була зосереджена саме в routes.py, та могла займати великий обсяг коду. Через те що валідація даних, звернення до бази даних, перевірки, формування відповіді відбувалось в одному файлі.

У лабораторній роботі 2 проєкт розділено на 4 окремі шари:

* Domain - доменні моделі (User, Sensor, Telemetry), Value Objects (Email, MacAddress), інтерфейси репозиторіїв, фабрики та доменні помилки. 

* Application - Use Cases, DTO, dependency injection.

* Infrastructure - реалізація репозиторіїв, ORM-моделі, реалізація інтерфейсів.

* Presentation - Наші роути але логіка викликається у більшості, з інших файлів. Розбирає HTTP запит, формує DTO, викликає Use Case, мапить доменні помилки в HTTP статуси.

ORM-модели (UserModel, LocationModel...) тепер повністю відокремлені від доменних моделей (User, Location...). Маппінг між ними відбувається в інфраструктурному шарі.

## Які переваги дало розділення на шари?

Можливість тестувати без підняття БД. Доменні моделі та фабрики - це чистий Python без залежностей від Flask чи SQLAlchemy. Їх можна тестувати без підняття бази даних або HTTP-сервера. 

Також кожен файл має одну зону відповідальності, що спрощує написання коду. Тепер повинно бути набагато простіше масштабуватись, бо зміна в одному шарі не зачіпає інші, тому якщо б над проєктом працювало багато розробників, на мою думку, було б простіше розподілити ролі.

## Які недоліки / ускладнення з'явилися?

Найбільше ускладнення, це те що потрібно було написати великий обсяг коду, щоб все працювало як потрібно. Для кожної операції тепер існує DTO, Use Case, метод в інтерфейсі репозиторію, реалізація в інфраструктурі та роут. Функціонал не змінився, але кодова база суттєво розширилась. Для подібного проєкту котрий не планує масштабуватись - це може бути трохи оверкомітнинг. 

Також для людини без знань про шарову архітектуру цей проєкт значно складніше, ніж якщо просто прочитати routes.py, models.py та розібратись. 

## Наскільки простіше тепер змінити БД або фреймворк?

Зміна БД стала простішою, всі SQL-операції знаходяться в infrastructure/repositories.py. За для переходу на іншу БД, достатньо просто написати нові реализації репозиторіїв, котрі будуть реалізувати корректно інтерфейси.

Зміна фреймворку - в теорії повинна буде простіше, якщо не змінювати ЯП, а наприклад просто змінити з Flask на FastAPI, достатньо буде переписати тільки рути, котрі в нас у presentation layer. 

## Чому було обрано Anemic / Rich Domain Model?

Я обрав Anemic Domain Model, через те що я не бачу сенса у Rich Domain Model. Мої сутності не мають складної власної поведінки, котру можна було б вкласти в доменну сутність. Водночас Value Objects (Email, MacAddress) містять валідацію в __post_init__, що є елементом Rich підходу - об'єкт сам захищає свої інваріанти і не може
існувати в невалідному стані, але вся основна логіка зосереджена в фабриках та юз кейсах.
