# Лабораторна робота 3: CQS (Command-Query Separation)

## Що змінилося в структурі проєкту порівняно з лабораторною 2?

Порівняно з лабораторною 2, першим чином, я розділив dto.py, на два файла commands.py та queries.py. Суть в тому, що Commands описують намір змінити стан системи, а Queries - параметри для отримання даних без побічних ефектів.
Після цього я перетворив мій use_cases.py на handlers.py, загалом концепт був такий самий, але команди не повинні повертати доменні моделі. Як у прикладі з LoginCommandHandler, де раніше я повертав User, а тепер лише user_id(допустимий компроміс для ідентифікації). Контролери стали ще менше вони лише маплять HTTP-запит у команду чи запит і передають відповідному хендлеру.

Також для того щоб було простіше тестувати API, я створив в docs/api_test.http; За допомогою його, навіть не використовуючи Postman, можна зручніше тестувати проєкт параллельно з написанням коду. Щоб його використовувати потрібно завантажити extension "REST CLIENT" у VS Code.
 
## Переваги CQS: чим допомогло розділення?

Кожен обробник (Handler) тепер має лише одну причину для зміни та мінімальну кількість залежностей, необхідних лише для його задачі. 
Також за допомогою ITelemetryReadRepository, можна робити денормалізовані запити до БД, не навантажуючи доменну логіку. Також через все це, ми можемо
тестувати Unit-тести швидше, за допомогою In-memory fakes. Ще можу додати, що через те що в нас розділено на Commands/Queries, то тепер простіше зрозуміти, який виклик(Command), може змінити дані, а який ні(Query).

## Недоліки CQS: що ускладнилося?

Як й в минулій лабораторній роботі, збільшилась кількість коду, через те що логіка роздроблена по багатьом дрібним файлам. Але в якійсь степені це може покращити читабельність, якщо людина розбирається в архітектурі. Наприклад, *GetTelemetryHistoryQueryHandler* тепер потребує окремої реалізації *SqlAlchemyTelemetryReadRepository* з JOIN-запитом - це додаткова інфраструктура якої не було в лабі 2.

## Чим підхід з Command / Query Handler відрізняється від Service, який робить все сам?

Handler - це вузькоспеціалізований клас, який отримує через конструктор тільки те, що йому потрібно для виконання однієї конкретної бізнес-операції. Чимось схоже на принцип UNIX-систем, інструмент має робити одну річ, але дуже вдало. Service же збирає величезну кількість залежностей, навіть якщо методу вистачить лише одна.

## Як CQS впливає на розширюваність? (додати нову команду vs змінити існуючий сервіс)

З CQS щоб додати нову фічу, нам достатньо створити нову команду та новий хендлер, не чіпаючи існуючий код. У випадку з великим сервісом, ми б змушені були редагувати вже існуючий клас, що збільшує ризик поламати стару логіку. Тобто впливає позитивно з точки зору безпеки, та зручності написання коду, але потрібно буде писати трохи більше коду, але це компенсуєтся тим, що не потрібно шукати в величезному сервісі, де й як, що змінити.

## Чи відрізняється структура даних, яку повертає Query, від доменної моделі? Чому це важливо?

Так, Query повертає Read Models, а не доменні агрегати. Це важливо через те що ми не передаємо структуру нашої бази даних назовні, що безпечніше. Якщо б ми передавали всі поля, а не тільки які потрібно клієнту, могли б бути витоки даних.
